## 一、项目概述

### 1.1 项目目标

RepoInspector 是一个 GitHub 仓库健康度分析工具。用户在 Web 页面输入一个公开 GitHub 仓库的 URL，系统通过 GitHub 官方 API 拉取仓库相关元数据，进行多维度统计分析，并以图表、卡片等可视化方式呈现结果，帮助用户快速评估一个开源项目的活跃度、质量和社区健康度。

### 1.2 核心价值

- **快速洞察**：在数秒内完成对一个仓库的全方位"体检"。
- **可视化呈现**：相比 GitHub 原始页面，提供更直观、更聚合的指标展示。
- **零门槛使用**：无需登录 GitHub、无需安装客户端，浏览器打开即用。

### 1.3 适用范围

- 公开仓库（public repo）均可分析。
- 单次分析一个仓库，不支持批量。
- 不修改任何仓库数据，只读访问。

## 三、技术架构

### 3.1 整体架构

采用 **前后端分离 + BFF（轻量后端）** 模式：

```
┌──────────────┐      HTTP/JSON       ┌──────────────────┐      HTTPS/JSON      ┌────────────────┐
│  前端浏览器  │  ◀──────────────▶   │  Python 后端     │  ◀──────────────▶   │  GitHub REST   │
│  (静态页面)  │                      │  (Flask/FastAPI) │                     │  API v3        │
└──────────────┘                      └──────────────────┘                     └────────────────┘
                                            │
                                            ▼
                                     ┌──────────────────┐
                                     │  内存缓存        │
                                     │  (LRU, 5min TTL) │
                                     └──────────────────┘
```

### 3.2 技术选型

| 层 | 技术 | 选型理由 |
|----|------|----------|
| 前端 | 原生 HTML5 + CSS3 + JavaScript (ES6+) | 用户硬性要求；无构建依赖，开箱即用 |
| 前端可视化 | **Chart.js 4.x**（CDN 引入） | 轻量、API 简单、社区成熟、支持响应式 |
| 前端 HTTP | Fetch API | 浏览器原生，无需额外库 |
| 后端框架 | **FastAPI** | 异步性能好、自动生成 OpenAPI 文档、原生支持 Pydantic 校验 |
| 后端语言 | Python 3.10+ | 用户硬性要求；类型注解友好 |
| HTTP 客户端 | **httpx** | 同步/异步统一，与 FastAPI 配合好 |
| 缓存 | 内存 dict + TTL（可选 Redis） | 单机部署场景下内存缓存足够 |
| 部署 | uvicorn + Nginx（生产）/ 直接 uvicorn（开发） | 简单可靠 |

### 5.1 配置管理（config.py）

```python
    request_timeout: float = 10.0

    class Config:

settings = Settings()
```

### 5.2 数据模型（models.py）

使用 Pydantic 定义请求/响应结构，便于自动校验与文档生成。

```python
from pydantic import BaseModel, Field
from typing import List, Optional

class AnalyzeRequest(BaseModel):
    url: str = Field(..., description="GitHub 仓库 URL，如 https://github.com/owner/repo")

class LanguageStat(BaseModel):
    name: str
    bytes: int
    percentage: float

class ContributorStat(BaseModel):
    login: str
    avatar_url: str
    contributions: int

class IssueStats(BaseModel):
    open: int
    closed: int
    avg_close_hours: Optional[float]

class MonthlyCommit(BaseModel):
    month: str          # "2025-07"
    count: int

class HealthScore(BaseModel):
    total: int                  # 0-100
    popularity: int             # 0-100
    activity: int               # 0-100
    community: int              # 0-100
    maintenance: int            # 0-100

class RepoReport(BaseModel):
    basic: dict                 # name, full_name, description, html_url, ...
    social: dict                # stars, forks, watchers, open_issues
    languages: List[LanguageStat]
    contributors: List[ContributorStat]
    commits_monthly: List[MonthlyCommit]
    issues: IssueStats
    releases: List[dict]
    health: HealthScore
    generated_at: str           # ISO 时间戳
```

### 5.3 GitHub 客户端（github_client.py）

封装对 GitHub API 的调用，处理分页、限流、错误。

**关键函数**：

| 函数 | 作用 | 端点 |
|------|------|------|
| `parse_repo_url(url)` | 解析出 owner/repo | - |
| `get_repo(owner, repo)` | 仓库基础信息 | `GET /repos/{owner}/{repo}` |
| `get_languages(owner, repo)` | 语言字节数 | `GET /repos/{owner}/{repo}/languages` |
| `get_contributors(owner, repo, per_page=30)` | 贡献者 | `GET /repos/{owner}/{repo}/contributors` |
| `get_commit_activity(owner, repo)` | 近 1 年 commit 统计（按周） | `GET /repos/{owner}/{repo}/stats/commit_activity` |
| `get_issues_count(owner, repo, state)` | Issue 数量 | `GET /search/issues` |
| `list_releases(owner, repo, per_page=5)` | Release 列表 | `GET /repos/{owner}/{repo}/releases` |

**限流处理**：
- 监听响应头 `X-RateLimit-Remaining`。
- 当 ≤ 10 时打印警告；= 0 时抛出 `RateLimitError`，返回 429 给前端。
- 401/403/404 等错误统一封装为 `GitHubAPIError`，包含状态码与 message。

**并发策略**：
- 多个独立请求使用 `asyncio.gather` 并发执行，控制并发数 ≤ 6。
- 单仓库总耗时应控制在 3 秒内。

### 5.5 缓存（cache.py）

```python
import time
from typing import Any

_cache: dict[str, tuple[float, Any]] = {}

def get(key: str) -> Any | None:
    if key in _cache:
        ts, val = _cache[key]
        if time.time() - ts < settings.cache_ttl_seconds:
            return val
        _cache.pop(key, None)
    return None

def set(key: str, val: Any):
    _cache[key] = (time.time(), val)
```

### 5.6 路由（routers/analyze.py）

```
POST /api/analyze
  Body: { "url": "https://github.com/facebook/react" }
  200: RepoReport
  400: { error: "invalid_url" }
  404: { error: "repo_not_found" }
  429: { error: "rate_limited", retry_after: 60 }

  200: { "status": "ok" }
```

主路由伪代码：

```python
@router.post("/analyze", response_model=RepoReport)
async def analyze(req: AnalyzeRequest):
    owner, repo = parse_repo_url(req.url)
    cache_key = f"{owner}/{repo}"
    cached = cache.get(cache_key)
    if cached:
        return cached

    # 并发拉取 6 类数据
    basic, langs, contribs, activity, issues, releases = await asyncio.gather(
        gh.get_repo(owner, repo),
        gh.get_languages(owner, repo),
        gh.get_contributors(owner, repo),
        gh.get_commit_activity(owner, repo),
        gh.get_issues_count(owner, repo, "all"),
        gh.list_releases(owner, repo),
    )

    report = RepoReport(
        basic=basic, social={...},
        languages=analyze_languages(langs),
        contributors=contribs[:10],
        commits_monthly=aggregate_monthly(activity),
        issues=analyze_issues(issues),
        releases=releases,
        health=compute_health_score(...),
        generated_at=now_iso(),
    )
    cache.set(cache_key, report)
    return report
```

## 七、数据流

### 7.1 正常流程

```
1. 用户在输入框输入 URL，点击"分析"
2. 前端校验 URL 格式
3. 前端 POST /api/analyze { url }
4. 后端解析出 owner/repo
5. 查缓存；命中则直接返回
6. 未命中则并发调用 GitHub API（6 个请求）
7. 原始数据 → analyzer → RepoReport
8. 写入缓存，返回 JSON
9. 前端拿到数据 → 渲染基础信息 + 5 个图表
```

### 7.2 异常流程

| 异常 | 处理 |
|------|------|
| URL 格式错误 | 前端拦截 + Toast 提示 |
| 仓库不存在（404） | 后端返回 404，前端展示"仓库不存在或为私有" |
| API 限流（403/429） | 后端返回 429 + retry_after，前端展示"请求过快，请稍后再试" |
| 网络错误 | 前端展示"网络异常，请检查连接" |
| 后端 5xx | 前端展示"服务异常，请稍后再试" |

## 八、关键算法说明

### 8.1 健康度评分

采用 **加权求和**（满分 100）：

| 维度 | 权重 | 评分依据 |
|------|------|----------|
| 流行度 | 30% | `log10(stars+1) / log10(200000) * 100`，超过 200k 视为满分 |
| 活跃度 | 30% | 近 3 月 commit 数 × 系数 + 近 1 年 Release 数 × 系数 |
| 社区 | 20% | 贡献者数量归一化 + Issue 关闭率（closed/(open+closed)） |
| 维护性 | 20% | 未 archived（+50）+ 存在 LICENSE（+25）+ 平均 Issue 解决 < 7 天（+25） |

最终分数按上述权重求和后取整。

> 注：评分为启发式估算，用于辅助参考，不代表仓库实际质量。

### 8.2 语言百分比

直接基于 GitHub API 返回的字节数：

```
percentage = lang_bytes / total_bytes * 100
```

保留 2 位小数，按百分比降序展示。

## 九、安全设计

1. **Token 保护**：GitHub Token 仅放在后端 `.env` 文件中，绝不暴露到前端或 Git 仓库。
2. **输入校验**：URL 严格正则校验，防止 SSRF 或注入。
3. **CORS**：生产环境仅允许同源；开发环境允许 `localhost`。
4. **日志脱敏**：不记录用户输入的 URL 明文到日志（可选）。
5. **依赖锁定**：`requirements.txt` 固定版本，定期更新。
